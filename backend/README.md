# Backend — Testing & Coverage

## Running tests

All commands run inside the Docker container:

```bash
docker compose exec backend pytest
```

This single command runs the full test suite, measures code coverage, and writes two report files consumed by SonarQube.

---

## How pytest-cov works

`pytest-cov` is a pytest plugin that wraps Python's built-in `coverage.py` library. When pytest starts, the plugin instructs `coverage.py` to begin tracing every line of source code that gets executed during the test run. Once all tests finish, it aggregates the trace data and writes the configured report formats.

### What gets measured

The `--cov=.` flag tells the plugin to measure coverage for every Python file found under the working directory (the `backend/` folder). The following paths are excluded via `--cov-omit` because they are either generated code or infrastructure files that have no meaningful test surface:

| Excluded path | Reason |
|---|---|
| `*/migrations/*` | Auto-generated by Django, not hand-written logic |
| `manage.py` | Django CLI entry point |
| `core/wsgi.py` | WSGI server hook, not application logic |
| `core/asgi.py` | ASGI server hook, not application logic |

## Generated reports

Every `pytest` run writes two files into the `reports/` directory **inside the container** (mapped to `backend/reports/` in the repo):

```
backend/
└── reports/
    ├── coverage.xml      ← Cobertura XML — code coverage per file and line
    └── test-results.xml  ← JUnit XML — pass/fail result for every test case
```

### `coverage.xml` (Cobertura format)

Generated by `--cov-report=xml:reports/coverage.xml`.

Contains one `<class>` entry per source file with:
- `line-rate` — fraction of lines covered (0.0 – 1.0)
- `<lines>` — every executable line marked as `hits="1"` (covered) or `hits="0"` (missed)

SonarQube reads this file to populate the **Coverage** metric on the dashboard. Point it at this file via:

```properties
sonar.python.coverage.reportPaths=backend/reports/coverage.xml
```

### `test-results.xml` (JUnit format)

Generated by `--junit-xml=reports/test-results.xml`.

Contains one `<testcase>` element per test with its name, class, duration, and failure details when applicable. SonarQube reads this to populate the **Tests** metric. Point it at this file via:

```properties
sonar.python.xunit.reportPath=backend/reports/test-results.xml
```

## pytest.ini reference

```ini
[pytest]
DJANGO_SETTINGS_MODULE = core.settings       # Django app settings loaded before tests run
python_files = tests.py test_*.py *_tests.py # File name patterns pytest collects

addopts =
    --junit-xml=reports/test-results.xml     # Write JUnit XML test results
    --cov=.                                  # Measure coverage for all files under backend/
    --cov-report=xml:reports/coverage.xml    # Write Cobertura XML coverage report
    --cov-omit=*/migrations/*,manage.py,core/wsgi.py,core/asgi.py
```

## Path note for SonarQube on Windows

The `coverage.xml` file contains absolute Linux paths from inside the container (e.g. `/app/users/models.py`). If SonarQube runs on Windows and the source root differs, configure a source path mapping in `sonar-project.properties` so SonarQube can resolve the paths correctly:

```properties
sonar.sources=backend
sonar.python.coverage.reportPaths=backend/reports/coverage.xml
sonar.python.xunit.reportPath=backend/reports/test-results.xml
```
